1 Android Crack: Injection, Hook, Pack

1.1 Xposed

通过改写 appprocess 和修改 clazz->method 为自定义的 native function 对任意 java 函数进行 hook.

xposed 的启动过程:

  1. 修改了 appprocess, 在启动 appprocess 时做以下几件事:
    1. 注册 XposedBridge 需要使用的 native 函数(通过 registerderobvandroidxposedXposedBridge), 主要的要注册的函 数是 hookMethodNative
    2. 通过 addXposedToClasspath 把 XposedBridge.jar 加入到 classPath 中
    3. 通过 dalvik 启动 XposedBridge.main, 而不是 ZygoteInit

XposedBridge.main 主要是做以下几件事:

  1. initXbridgeZygote, 这个函数的主要功能是 hook handleBindApplication 这个函数, 在应用启动时就会调用 XposedBridge 定义的相应的 hook, 进而 调用 XCLoadPackage.callAll(lpparam), 后者会对所有注册的 Xposed 模 块调用相应的 callback.
  2. loadModules,通过扫描 xposed 自身维护的 conf/module.list 扫描所有已 安装的模块, 然后将各个模块中定义的 callback 加入到上一步提到的 XCLoadPackage 中.
  3. 调用原始的 ZygoteInit

各个模块的 callback 在应用启动时被调用, 相应的 callback 函数会进一步通 过 findAndHookMethod 来 hook 更多的函数.

1.1.1 findAndHookMethod

xposed 的核心是 findAndHookMethod 函数.

这个函数最终会调用到 xposed framework 中定义的 hookMethodNative 中, hookMethodNative 会调用以下函数: 

method->nativeFunc = &xposedCallHandler;
method->insns = (const u2*) hookInfo;

即, 将 method 的 nativeFunc 强制指向 xposedCallHandler, 使该函数变成一 个 native 函数, 同时将 method 无用的 insns 指向一个自定义的 hookInfo结 构体, 这个结构会保存原始的函数的信息和 hook 的信息.

xposedCallHandler 做为所有被 hook 函数统一的入口, 做的事主要是调用 XposedBridge 中定义的 handleHookedMethod, 这个函数会调用用户自定义的各 个 hook 方法, 包括 beforeHookedMethod, afterHookedMethod, 以及原始的方 法.

1.2 Cydia substrate

不确定如何工作的, 但据说是通过 ptrace 进行 so injection 后修改 clazz->method 为自定义 native function 实现的.

1.3 libinject

通过 ptrace 进行 so injection.

so injection 是其它形式的 hook 的第一步:

  1. 若要 hook 其它 so 中的函数, 例如 open 调用, 可以通过 so injection 后再 目标进程中修改 elf 的 GOT 来完成 hook
  2. 若要 hook 某些 java 调用, 可以通过 so injection 后通过修改一些 dalvik 相关的数据来完成 hook, 例如: 直接修改某个 method 的 inst, 或 者修改某些对象的引用来自定义的对象等

1.4 梆梆加固

对 apk 进行加壳 Andorid APK反逆向解决方案：梆梆加固原理探寻 AndroidDynamicLoader

原理:

  1. 大部分组件, 例如 Application, Activity, Receiver, Service 都是通过 LoadedApk.getClassLoader 获得的 ClassLoader 来加载的, 这个 ClassLoader 代表了实际的 apk. 但 Provider 比较另类: 如果 Application 类被重写了, 则它优化使用 Application.getClassLoader 来 加载.
  2. 对于 Provider, 可以简单的通过重写一个 Application 类来实现
  3. 对于其它组件, 因为都是 LoadedApk 来加载的, 而 LoadedApk 在 Application 中有一个引用: Application.mBase.mPackageInfo, 所以我们 可以通过反射修改 Application.mBase.mPackageInfo.mClassLoader 来实现.

因为 Application 初始化比其他任何组件都早, 所以在 Application 实现上述 做法是很好的方式.

具体的:

  1. 加固程序会生成一个新的 apk, 这个 apk 的 manifest 与原 apk 类似, 也包 含原有的组件声明, 但会修改原程序的 manifest, 填加或修改一个 Application 类, 这个类是解壳时的起点.
  2. 新的 apk 的 dex 中会有一块区域嵌入了经过加密的原 odex 文件, 相当于 新的 dex 是原 odex 上加了一个壳.
  3. 当 ActivityThread 加载新 apk 的 Application 类时, 这个类的代码会负 责将原 odex 解密到内存, 然后调用 c++ 层的 dexFileParse 等函数生成 DexFile, 最终通过 jni 生成一个 PathClassLoader
  4. 通过反射等方法用这个 ClassLoader 代替 LoadedApk.mClassLoader, 后续 加载其他组件时都会用这个 ClassLoader.
